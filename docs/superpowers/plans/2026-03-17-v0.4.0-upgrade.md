- [ ] **Step 6: Remove forwardRef from all components**

This is a large mechanical change across 65 components. The pattern transformation is:

**Before (React 18 with forwardRef):**
```typescript
export const Button = React.forwardRef<HTMLButtonElement, ButtonProps>(
  ({ appearance = 'outline', size = 'medium', className, children, ...props }, ref) => {
    return (
      <button ref={ref} className={cn(...)} {...props}>
        {children}
      </button>
    );
  },
);
Button.displayName = 'Button';
```

**After (React 19 with ref as prop) — standard components:**
```typescript
export const Button = ({ appearance = 'outline', size = 'medium', className, children, ref, ...props }: ButtonProps & { ref?: React.Ref<HTMLButtonElement> }) => {
  return (
    <button ref={ref} className={cn(...)} {...props}>
      {children}
    </button>
  );
};
Button.displayName = 'Button';
```

**After — polymorphic components with `as` prop:**
Components that accept an `as` prop (Button, CompoundButton, MenuButton, SplitButton, ToggleButton, Link, and potentially others) render a dynamic element type. For these, use a broader ref type:
```typescript
export const Button = ({ appearance = 'outline', size = 'medium', as, className, children, ref, ...props }: ButtonProps & { ref?: React.Ref<HTMLElement> }) => {
  const Component = as || 'button';
  return (
    <Component ref={ref} className={cn(...)} {...props}>
      {children}
    </Component>
  );
};
```
Using `React.Ref<HTMLElement>` instead of `React.Ref<HTMLButtonElement>` accommodates the dynamic element type. This is a pragmatic trade-off — the ref type is slightly less specific but avoids complex generic type gymnastics.

Key changes per component:
1. Remove `React.forwardRef<ElementType, PropsType>(...)` wrapper
2. Move `ref` from second argument into destructured props
3. Add `ref?: React.Ref<ElementType>` to the props type (either inline or in the interface). For polymorphic components with `as` prop, use `React.Ref<HTMLElement>`.
4. Keep `displayName` assignment

For **compound components** using `Object.assign` (e.g., `Card.tsx`), the same transformation applies to each sub-component. The `Object.assign` pattern does not change.

## Troubleshooting Reference

### If `vite-plugin-dts` is incompatible with Vite 8
Option A: Pin to last compatible version and open an issue upstream.
Option B: Remove `vite-plugin-dts` from `vite.config.ts` plugins and update the `build` script in `package.json` to emit declarations separately:
```json
"build": "tsc --noEmit && vite build && tsc --emitDeclarationOnly --declaration --outDir dist"
```
This ensures `dist/index.d.ts` is still generated as part of `npm run build`.

### If `vitest-axe` is incompatible with Vitest 4
Option A: Check if a fork or newer version exists on npm.
Option B: Remove `vitest-axe`, use `@axe-core/react` or inline axe checks in tests.

### If Tailwind 4 class names changed
Tailwind 4 renamed some utilities. Check the Tailwind 4 migration guide. Common renames:
- `bg-opacity-*` → `bg-*/opacity` syntax
- `text-opacity-*` → similar
- Some spacing utilities may have changed

Run `npm run build` and check for Tailwind warnings about unknown utilities.

### If Storybook migration CLI fails
Fall back to manual migration:
1. Update all `@storybook/*` packages to `^10.2.0` in package.json
2. Run `npm install`
3. Check Storybook 10 migration guide for config changes
4. Update `.storybook/main.ts` and `.storybook/preview.tsx` manually
